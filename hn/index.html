Run-Length Encoding for arbitrary-length substrings


Try it out!


FAQ

What is this?

Compressing messages while still making it readable to a human.

{...not that special...}

A more interesting example would be "yolo yolo yolo baggins yolo baggins yolo baggins". A naive implementation may compress the starting yolo's and are forced to compress the rest like "<yolo x3> <baggins yolo x2> baggins". However, it identifies that the compression "<yolo x2> <yolo baggins x3>" is more optimal. But it's not perfect. Try "yolo yolo yolo baggins yolo baggins". Optimal would be "<yolo x2> <yolo baggins x2>" but that's not the case.

How does it work?

It uses suffix arrays to create a set of whole-word substrings that are repeated more than once. Then the algorithm poses the compression as a set-packing problem and optimizes for a minimum length of the compressed message. Length is defined as the sum of the size of the subset of patterns chosen. The algorithm uses only a heuristic to find the minimum length. I have experimented with using a linear programming solver but from what I tried, I would have to enumerate all possible "compressions" given a pattern to feed into the solver. For example, given "yolo yolo yolo" and given that "yolo" is identified as a repeated pattern by the suffix array, I would have to enumerate:

  • yolo yolo yolo yolo
  • <yolo x2> yolo yolo
  • <yolo x3> yolo
  • <yolo x4>
  • yolo <yolo x2> yolo
  • yolo <yolo x3>
  • yolo yolo <yolo x2>

Is this "sentence-compression"?

We are certainly compressing messages because we're reducing the length of a sentence while maintaining its meaning. However, the algorithms do not look at any semantics of the original message. When we reduce repeats to just one utterance, we just happen to retain the meaning because it's being uttered more than once.

Is this "data-compression"?

Yes but at a much more simplistic level. Advanced techniques of compression prioritize data-size (of course) which is extremely beneficial for storage and transfer but not for human-consumption. So from a data-compression perspective, we aim for an encoding scheme that is easily and quickly decoded by a human.
